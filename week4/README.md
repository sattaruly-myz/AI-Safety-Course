# Answer to the questions

---

## Assignment 1 (modular_arithmetic tool test)

Изначально маленькая модель пыталась считать остаток от деления «в голове», не вызывая инструмент, из-за чего часто ошибалась на больших числах.

Чтобы это исправить, я сделал промпт максимально жёстким:
*“You MUST use the modular_arithmetic tool…”*

Это принудило модель использовать инструмент вместо внутреннего «угадывания».

**Вывод:** маленькие модели очень чувствительны к формулировкам промпта, особенно когда речь идёт о вызове инструментов.

---

## Assignment 3a — Approach 0 (без инструментов)

Модель допускала серьёзные арифметические ошибки на больших числах (например, при умножении).

Причина — в архитектуре LLM:

* числа разбиваются на токены, теряя структуру;
* нет внутренней «рабочей памяти» для переноса разрядов.

---

## Assignment 3b — Approach A (наивный tool loop)

Практически не дал улучшений.

Модель либо игнорировала инструменты, либо пыталась вызвать инструмент и сразу же выдать финальный ответ, не дожидаясь результата.

**Вывод:** без чёткой структуры «вызвал → получил результат → продолжил» инструменты почти бесполезны.

---

## Assignment 3c — Сравнение подходов

ReAct-подход дал наилучшую точность, но оказался самым дорогим (по числу сообщений и задержке).

Разница в поведении:

* наивный loop — одна длинная генерация;
* ReAct — пошаговый процесс: *Thought → Action → Observation*.

Такая структура не даёт модели «зафиксироваться» на ошибочном ответе слишком рано.

## Assignment 2 (sympy_solve tool test)

Сначала модель пыталась решать уравнения сама и делала ошибки (например, со знаками).

Я изменил промпт:
*“Do not solve the equation yourself. Pass it to sympy_solve.”*

Это заставило модель полностью делегировать вычисления инструменту.

## Assignment 3 (math scorer test)

Модель-судья корректно распознала, что 10/4 и 5/2 — это одно и то же.

**Проблемы LLM-as-a-judge:**

* **False Negatives:** слишком строгая проверка формата (разные, но эквивалентные записи)
* **False Positives:** модель может «поверить» в убедительное, но неверное объяснение

## Assignment 4 (Dev-Set Iteration)

**Изменения:**

* ужесточил system prompt
* явно указал, какой инструмент когда использовать

**Самый большой эффект:**
Исправление формата `submit()`

Ранее модель решала задачу правильно, но не проходила проверку из-за формата — это типичный случай «ложной ошибки» из-за парсинга.

**Что ухудшало результат:**
Слишком много инструментов → модель путалась (analysis paralysis)

**Типичные ошибки:**

* неправильный выбор инструмента
* преждевременная остановка

---

## Assignment 5 (Test-Set Evaluation)

**Разрыв dev vs test:**
Точность на тесте немного ниже → лёгкое переобучение на dev (optimization bias)

**По темам:**

* хорошо: простая алгебра (Level 1–2)
* плохо: сложные темы (теория чисел, сложная алгебра)

**Как улучшить:**
Заменить набор простых инструментов на `python_repl`

Python даёт:

* циклы
* сложную логику
* гибкость для решения нетривиальных задач

## Bonus — Error Analysis

**Самая частая ошибка:**
Ошибки рассуждения — модель правильно считает шаги, но строит неверный план решения

**Как исправлять:**

* ошибки формата → через промпт и структуру
* сложные задачи → через более мощные инструменты (Python)
* ошибки логики → только более сильная модель

**Ошибки судьи:**
Да, судья иногда не распознаёт эквивалентные сложные выражения

**Вывод:**
Модель-судья должна быть заметно сильнее оцениваемой модели, иначе оценка становится шумной и ненадёжной.
